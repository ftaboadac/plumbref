**Strongest Trust Problems**

1. False confidence can enter through omitted claims. Plumbref has no independent answer coverage check. `plumbref_extract_claims` stores whatever claims the agent supplies, and the README admits claim extraction is agent work. If the agent leaves out the riskiest sentence, Plumbref cannot know the answer was incompletely checked. See [mcp_server.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/mcp_server.py:100).

2. The “gate” is not a hard reliance block. When supported claims miss required template checks, `build_answer_gate` returns `answer_with_qualifications` with `can_answer: True`, not a block. That may be reasonable product behavior, but it is weaker than “cannot be marked safe to rely on unless the required evidence trail is present.” See [reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/reports.py:680).

3. `verdict` can contradict the answer gate. `build_json_report` sets `verdict` from raw claim statuses before strict gate coverage. A report with all claims marked `supported` but missing required searches can have `verdict: Supported` while the quality gate says “Answer with qualifications.” That is a direct false-confidence surface for integrations reading JSON. See [reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/reports.py:166) and [reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/reports.py:1362).

4. Contradiction checking is mostly attestation. A supported judgment needs `contradiction_searched=True`, but Plumbref does not prove the contradiction search was sufficient unless a selected template happens to require matching query tokens. For broad claims, `contradiction_notes` only needs text. See [models.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/models.py:281) and [judgments.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/judgments.py:23).

5. Template checks are syntactic. Required searches pass when normalized tokens from the template pattern appear in recorded query text and, for required searches, some matched file exists. This does not ensure the agent searched the right symbol scope, callers, imports, alternate names, generated files, or config paths. See [reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/reports.py:906).

6. Evidence categories are easy to overstate. Only `tests` and `docs` get simple path checks; all other categories return `True`. An agent can tag a snippet as “direct consumers” or “external integrations” without Plumbref verifying that category semantically. See [reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/reports.py:931).

7. Required template placeholders can be skipped as “not applicable.” `resolve_pattern` treats values like `none`, `not applicable`, and `no external system` as skipped/passed. That can be correct, but there is no required justification or evidence for why a category is genuinely not applicable. See [reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/reports.py:882).

8. Budgets are enforced as limits, not as proof of adequate search. Budget exhaustion is recorded and downgrades safe output, which is good. But budgets do not guarantee a minimum search depth unless templates are selected and populated correctly. Also, cache hits can produce compact evidence with no excerpt, reducing auditability in the Markdown report. See [budgets.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/budgets.py:5), [evidence.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/evidence.py:59), and [reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/reports.py:1355).

9. Example proof is not clean enough yet. The first-run transcript says it is a fixture, not a captured named-client transcript. Historical dogfood reports are explicitly stale. That supports “workflow shape,” not a public-grade verification proof. See [mcp-stdio-transcript.md](/Users/facundotaboada/Documents/GitHub/plumbref/examples/first-run/mcp-stdio-transcript.md:1) and [README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:274).

10. PR checking is too narrow for the proposed direction. The advisory claim checker detects “only docs/copy” versus changed code/config files, broad language, and docs-only plausibility. It does not inspect PR prose against code evidence, run templates, or enforce checked-claim schema in CI. See [claim_checks.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/claim_checks.py:91).

**Trust Boundary**

The real trust boundary is: Plumbref can verify that an agent recorded claims, searches, snippets, statuses, limits, and reports in a structured format. It cannot verify that the agent chose the right claims, chose exhaustive searches, interpreted source correctly, or covered unmentioned answer text. That boundary is honestly documented, but it materially narrows the verification claim.

would try once

The single artifact that would most change my mind: an adversarial strict-gate test suite and captured named-client transcript where a sloppy agent tries to mark broad claims supported while omitting claims, using weak searches, fake categories, “not applicable” placeholders, cache-only evidence, and budget-exhausted traces, and Plumbref reliably prevents `safe_to_answer` and prevents JSON `verdict: Supported`.
