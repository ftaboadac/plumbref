**Adoption Review**

The concrete moment that would make me run Plumbref is narrow: an agent says something broad before I merge, rename, brief support, or write a PR summary: “only affects onboarding,” “safe to rename,” “no downstream consumers,” “SSO only depends on Okta.” That moment is **partially proven** by the README examples and first-run fixture ([README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:18), [examples/first-run/README.md](/Users/facundotaboada/Documents/GitHub/plumbref/examples/first-run/README.md:9)). It is most compelling when Plumbref rejects an overclaim, like “only docs,” and gives safer wording plus source lines ([examples/reports/reliance-check-overclaim-demo.md](/Users/facundotaboada/Documents/GitHub/plumbref/examples/reports/reliance-check-overclaim-demo.md:19)).

The report is useful when it changes a decision: supported claims, contradicted claims, safer wording, evidence, and unchecked areas are concrete enough to be a receipt. That is **proven by repo/demo** for the overclaim examples. But many reports are still heavy and self-referential, especially the template-loading dogfood report; those prove the harness can document itself more than they prove I would use it on product code. Report usefulness as a normal daily artifact is **partially proven**, not proven.

The biggest trust objection is that Plumbref does not extract claims or decide truth independently. The repo is explicit about this: “The agent still extracts claims and reasons over evidence” ([README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:46)), and `plumbref_extract_claims` stores claims supplied by the agent ([plumbref/mcp_server.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/mcp_server.py:100)). The CLI `verify` path says the MVP does not extract claims automatically if no claims are supplied ([plumbref/cli.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/cli.py:253)). So the differentiation over “ask the agent to be careful and cite files” is **partially proven** by structure, but **unproven** as a behavior improvement in normal agent chats.

The strict gate is better than pure ceremony. Supported judgments require evidence and a contradiction pass ([plumbref/models.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/models.py:277)), broad supported claims require contradiction notes ([plumbref/judgments.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/judgments.py:23)), and tests assert missing required searches, unlinked evidence, and budget exhaustion downgrade safety ([tests/test_reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/tests/test_reports.py:502), [tests/test_reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/tests/test_reports.py:573), [tests/test_reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/tests/test_reports.py:620)). Claim: “Plumbref blocks unsupported confidence when required traces are missing” is **partially proven**. Claim: “Plumbref verifies AI coding-agent claims before users rely on them” is **too vague to evaluate** unless “verifies” means “forces a recorded workflow and downgrades incomplete traces.”

Friction is real. Install is simple enough with `pipx install plumbref`, but the actual product depends on MCP configuration and client-specific placement that the docs do not spell out beyond generic JSON ([README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:105), [docs/agent-usage.md](/Users/facundotaboada/Documents/GitHub/plumbref/docs/agent-usage.md:37)). The agent instruction block is long and discipline-heavy ([docs/agent-usage.md](/Users/facundotaboada/Documents/GitHub/plumbref/docs/agent-usage.md:91)). If the agent forgets to use the tool, chooses weak claims, records shallow searches, or summarizes evidence badly, Plumbref mostly exposes that rather than fixing it. That trust boundary is **proven by repo**.

I would use this inside an agent chat or not at all. The README says users should not manually run verification commands during normal use ([README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:79)), and the roadmap says CLI is not the product surface ([ROADMAP.md](/Users/facundotaboada/Documents/GitHub/plumbref/ROADMAP.md:8)). CLI is setup/debug. CI is currently too thin: `check-claims` parses an explicit claims file and compares risky wording to changed files; it is advisory and mostly catches obvious “only docs” contradictions ([plumbref/claim_checks.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/claim_checks.py:14)). CI/team adoption is **unproven** beyond a prototype.

What would make me uninstall after one try: no captured first-run transcript for Codex/Claude/Cursor, MCP setup ambiguity, the agent producing a long report without a better answer, or the first report downgrading everything because required template values/searches were missing. The repo admits the first-run artifact is still expected output, not a captured live client transcript ([examples/first-run/README.md](/Users/facundotaboada/Documents/GitHub/plumbref/examples/first-run/README.md:47), [docs/checked-claims-implementation-plan.md](/Users/facundotaboada/Documents/GitHub/plumbref/docs/checked-claims-implementation-plan.md:205)). “Works in a normal coding-agent workflow” is therefore **unproven**.

This is worth it only if my codebase is grep-friendly, source-local, and risk-heavy enough that unsupported agent claims cause real cost. It also requires a team willing to maintain templates/config and accept “supported by checked evidence” rather than global truth. It is not worth it for small repos, low-risk changes, teams already disciplined about `rg` plus cited line ranges, or workflows where production state/external systems matter more than repo evidence. These constraints are **proven by repo limitations** ([README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:321)).

The missing proof before I would recommend it: a public, reproducible transcript in one named client from install/MCP setup to final inline answer, on a non-self-referential fixture, compared against a careful prompt plus `rg`. The repo itself lists this benchmark as not started ([docs/checked-claims-implementation-plan.md](/Users/facundotaboada/Documents/GitHub/plumbref/docs/checked-claims-implementation-plan.md:22)).

**Verdict:** would try once.

Single artifact that would most change my mind: a captured Codex or Claude Code first-run transcript showing the agent initially making or checking a risky claim, Plumbref forcing missing checks or contradiction search, and the final answer materially improving over a careful `rg`-and-citations baseline.
