**Alternatives Review**

Verdict: **would try once**

Plumbref’s strongest real value is not that it finds things `rg` cannot find. It is that it makes the agent record claims, searches, snippets, judgments, limits, and safer wording in a repeatable artifact. That is **partially proven** by the MCP tools and reports: claims are agent-supplied, searches use ripgrep, evidence is bounded, supported judgments require evidence and a contradiction pass, and reports downgrade missing gate coverage ([mcp_server.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/mcp_server.py:100), [search.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/search.py:18), [models.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/models.py:267), [reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/reports.py:623)).

But against substitutes, the current case is still narrow.

For ordinary “where is this used?” or “what changed?” questions, `rg`, manual source reading, type checking, linters, tests, CI, and a code review checklist are cheaper and good enough. Plumbref itself depends on `rg` for search ([README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:127), [search.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/search.py:115)). It does not build a semantic index, call graph, type graph, or static analysis engine; those are still roadmap work ([ROADMAP.md](/Users/facundotaboada/Documents/GitHub/plumbref/ROADMAP.md:132)). Claim: **proven by repo**.

For correctness regressions, tests/CI/type checking/linters are better substitutes. Plumbref can say “this claim lacks evidence” or “this code path was checked,” but it cannot prove runtime behavior, production state, or external-system truth. The README admits repo-local lexical limits and no production-data inspection ([README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:321)). Claim: **proven by repo**.

For security, policy, framework, API misuse, and taint-style issues, Semgrep/static analysis is the better substitute. Plumbref’s templates are checklists with string searches such as `{changed_symbol} caller`, `{changed_symbol} import`, and `{field_name} external api webhook sync` ([change_impact.toml](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/templates/change_impact.toml:8), [field_migration.toml](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/templates/field_migration.toml:8)). That is useful discipline, not static analysis. Claim: **proven by repo**.

For teams already using Codex/Claude/Cursor carefully, the cheaper substitute is: “cite files and lines, search for contradictions, list unchecked areas, and do not answer globally from local snippets.” Plumbref’s own agent instructions are basically that workflow encoded as MCP steps ([docs/agent-usage.md](/Users/facundotaboada/Documents/GitHub/plumbref/docs/agent-usage.md:91)). The difference is that Plumbref records and gates the trace instead of trusting the agent to remember. Important, but only when the team does not trust voluntary discipline. Claim: **partially proven**.

The strict gate is the key differentiator over `AGENTS.md` or project rules. The current repo has made progress: supported judgments require evidence and contradiction search, required searches can downgrade safe output, unlinked evidence is caught, and budget-exhausted traces downgrade claims ([tests/test_reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/tests/test_reports.py:502), [tests/test_reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/tests/test_reports.py:573), [tests/test_reports.py](/Users/facundotaboada/Documents/GitHub/plumbref/tests/test_reports.py:620)). But per-claim evidence category semantics and benchmark proof are still explicitly unfinished ([docs/checked-claims-implementation-plan.md](/Users/facundotaboada/Documents/GitHub/plumbref/docs/checked-claims-implementation-plan.md:39), [docs/checked-claims-implementation-plan.md](/Users/facundotaboada/Documents/GitHub/plumbref/docs/checked-claims-implementation-plan.md:400)). Claim: **partially proven**.

Living checked claims and rerun/diff are the most differentiated substitute-resistant idea. The SSO drift diff is a good artifact: it shows a claim moving from `supported` to `contradicted` after fixture changes ([sso-eligibility-drift-diff.md](/Users/facundotaboada/Documents/GitHub/plumbref/examples/reports/sso-eligibility-drift-diff.md:19)). But the rerun command currently emits a packet telling an agent how to rerun, not an automatic rerun verifier ([checked_claims.py](/Users/facundotaboada/Documents/GitHub/plumbref/plumbref/checked_claims.py:193)). Claim: **partially proven**.

The README should explicitly acknowledge the strongest alternative: **careful prompting plus `rg`, file citations, contradiction searches, and unchecked-area reporting**. It currently mentions prompts and careful agents, but not as a first-class alternative with a “when this is enough” section ([README.md](/Users/facundotaboada/Documents/GitHub/plumbref/README.md:39)). Claim: **partially proven**.

The benchmark that would prove Plumbref is not just careful prompting plus `rg`: 20 realistic repo-claim tasks across unfamiliar repos, comparing a careful prompted agent, a Plumbref-guided agent, and human-reviewed ground truth, measuring unsupported final claims, missed contradiction paths, unchecked areas surfaced, time/tool calls, and auditability. This is already described but not started ([docs/checked-claims-implementation-plan.md](/Users/facundotaboada/Documents/GitHub/plumbref/docs/checked-claims-implementation-plan.md:411)). Claim: **unproven**.

Single artifact that would most change my mind: a public captured transcript from one named MCP client showing install/config, tool calls, generated report, and a case where Plumbref downgrades or blocks a claim that a careful prompted baseline with `rg` misses.
