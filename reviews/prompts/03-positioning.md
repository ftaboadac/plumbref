language is clear, narrow, credible, and motivating.

Do not read `reviews/out/`. Do not infer or summarize other reviewer opinions.
Use the repository and the supplied review packet. Cite files, README sections,
examples, and missing artifacts where possible.

words and claims that make the project sound inflated, vague, or easy to ignore.

1. Who is the actual user?

3. Which claim is too broad for the current proof?
4. What proof is missing from the README?
5. What is the clearest one-sentence product description supported by the repo?
6. What should be removed, narrowed, or moved lower?
7. Does the README create a concrete "I want this before changing risky code"

- proven by repo/demo
- partially proven
